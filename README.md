<h1 align="center">Meteor History</h1>

<p align="center">A self-hosted GitHub repository gallery with star history charts and GitHub profile cards.</p>

Meteor History periodically reads GitHub data in the background and stores the generated repository index, profile statistics, and star histories as JSON. Visitors only read the cached data; they do not start GitHub synchronization jobs.

## Features

- Personal repository gallery with stars and fork counts
- Star History-style SVG charts with light, dark, and automatic color schemes
- GitHub Readme Stats-style profile cards
- Copyable image URLs and Markdown snippets
- Background synchronization with JSON persistence
- Fine-grained GitHub token repository filtering
- Signed image URLs, rate limiting, hotlink protection, and redacted server errors
- Docker images for Linux AMD64 and ARM64

## Requirements

- Node.js 22 or newer for local development
- A fine-grained GitHub personal access token with Metadata read access to the repositories that should be displayed
- Docker Engine with Docker Compose for container deployment

## Local development

Copy the example environment file and provide your secrets:

```powershell
Copy-Item .env.example .env
npm ci
npm run dev
```

The Vite development server runs on port `5173`. The API port comes from `PORT` and defaults to `3000` when it is not set.

Run all tests and create a production frontend build with:

```powershell
npm run check
```

## Environment security

Never commit `.env`, GitHub tokens, signing keys, certificates, or cache files. The included `.gitignore` and `.dockerignore` exclude these files, and the Docker build does not copy runtime secrets into the image.

Production requires at least these values:

```dotenv
GITHUB_TOKEN=github_pat_xxx
EMBED_SIGNING_KEY=replace-with-a-unique-random-value-of-at-least-32-characters
PUBLIC_BASE_URL=https://meteor-history.com
```

Keep `INCLUDE_PRIVATE_REPOSITORIES=false` for public deployments. Store `.cache` on persistent storage because it contains the JSON generated by scheduled synchronization.

## Docker deployment

Create `.env` first, then build and start the service:

```powershell
docker compose up -d --build
```

The application is available at `http://localhost:8666`. The Compose configuration stores JSON data in the `meteor-history-cache` volume.

To use a published image without building locally:

```powershell
docker pull ghcr.io/mengmengcode/meteor-history:latest
docker compose up -d
```

Published version tags include:

- `ghcr.io/mengmengcode/meteor-history:1.2.3`
- `ghcr.io/mengmengcode/meteor-history:1.2`
- `ghcr.io/mengmengcode/meteor-history:1`
- `ghcr.io/mengmengcode/meteor-history:latest`

The `main` branch is published as `ghcr.io/mengmengcode/meteor-history:main`.

## Automated builds and releases

The workflow at `.github/workflows/build-release.yml` performs these operations:

1. Runs the test suite and production frontend build.
2. Builds the Docker image for Linux AMD64 and ARM64.
3. Publishes branch and semantic version tags to GitHub Container Registry.
4. Generates software bill of materials and provenance metadata for published images.
5. Creates a GitHub Release with automatically generated release notes for version tags.

Pull requests run validation and a local AMD64 container build without publishing an image. Pushes to `main` publish the `main` image tag. Tags matching `v1.2.3` publish versioned images and create a GitHub Release.

The workflow uses the repository-provided `GITHUB_TOKEN`. In repository settings, GitHub Actions must have permission to write packages and repository contents. The first GHCR package may be private by default; change its package visibility if anonymous pulls are required.

## One-command release

The release script requires a clean Git working tree, an `origin` remote, npm, Git, and Docker. Run it from PowerShell:

```powershell
.\scripts\release.ps1 -Version 1.2.3
```

The script performs the following operations:

1. Validates the Git repository, remote, clean working tree, and version tag.
2. Installs locked dependencies.
3. Updates `package.json` and `package-lock.json`.
4. Runs the complete project check.
5. Builds local `meteor-history:1.2.3` and `meteor-history:latest` images.
6. Creates a release commit and annotated `v1.2.3` tag.
7. Atomically pushes the branch and tag to `origin`.

The pushed tag starts the GitHub Actions release workflow. To publish without a local Docker build, use:

```powershell
.\scripts\release.ps1 -Version 1.2.3 -SkipDockerBuild
```

Use a different local image name or Git remote when required:

```powershell
.\scripts\release.ps1 -Version 1.2.3 -ImageName custom/meteor-history -Remote upstream
```

## Runtime data

Repository and profile metadata are stored under `.cache` as JSON. This directory is runtime state and is intentionally excluded from Git. In Docker it is mounted at `/app/.cache` through a named volume.
